# Отчёт по результатам анализа Kubernetes Audit Log

## Подозрительные события

1. Доступ к секретам:
    - Кто: system:apiserver
    - Где: namespace kube-system
    - Почему подозрительно:
      Сам по себе доступ apiserver является штатным поведением, однако наличие доступа к secrets требует контроля RBAC.
      При наличии аналогичных действий со стороны обычных пользователей это могло бы означать компрометацию кластера.

2. Привилегированные поды:
    - Кто: minikube-user
    - Где: namespace secure-ops
    - Что произошло:
      Создан pod privileged-pod с контейнером pwn, у которого выставлено securityContext.privileged=true.
    - Почему подозрительно:
      Привилегированный pod получает расширенные возможности на ноде и может использоваться для выхода из контейнера,
      доступа к хосту, кражи токенов и дальнейшего повышения привилегий.

3. Использование kubectl exec в чужом поде:
    - Кто: не обнаружено
    - Что делал:
      События exec отсутствуют.

4. Создание RoleBinding с правами cluster-admin:

    - Кто: minikube-user
    - Где: namespace secure-ops
    - Что произошло:
      Пользователь создал RoleBinding "escalate-binding"

    - К чему привело:
      Возможное повышение привилегий пользователя до уровня cluster-admin.
      Это является критическим событием безопасности.

5. Удаление audit-policy.yaml:

    - Кто: не обнаружено
    - Возможные последствия:
      Не зафиксировано попыток удаления audit policy.
    - Примечание: в ходе выполнения задания не удалось добиться выполнения команды на удаление audinig-policy.yaml.
      Полученное сообщение об ошибке: `error: resource mapping not found for name: "" namespace: "" from "/home/afjord/.minikube/files/etc/ssl/certs/audit-policy.yaml": no matches for kind "Policy" in version "audit.k8s.io/v1"
      ensure CRDs are installed first`

6. Доступ к clusterrolebindings:

    - Кто:
        - kubernetes-admin
        - kubernetes-super-admin
        - minikube-user

    - Комментарий:
      Управление clusterrolebindings потенциально позволяет изменить модель безопасности кластера.

## Возможная компрометация кластера

Создание RoleBinding escalate-binding пользователем minikube-user может свидетельствовать о попытке повышения
привилегий.

Это можно считать потенциальной компрометацией RBAC политики безопасности.

## Ошибки политики RBAC

Обнаружены следующие проблемы:

1. Пользователь minikube-user имеет возможность создавать RoleBinding.
2. Возможна эскалация привилегий через namespace secure-ops.
3. Отсутствует ограничение на создание bindings с повышенными правами.
4. Нет разделения административных и пользовательских ролей.

## Вывод

Наиболее критичным событием является создание RoleBinding escalate-binding пользователем minikube-user.

Это указывает на возможность повышения привилегий внутри кластера и свидетельствует о некорректной настройке RBAC
политики безопасности.
