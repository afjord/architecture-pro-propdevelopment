## Требования к внешним интеграциям для сервисов «Умный дом»

### 1. Требования к безопасности

1. Все взаимодействия между системами PropDevelopment и внешней платформой умного дома должны выполняться по защищённым каналам связи с использованием
   **TLS 1.2+**, предпочтительно **TLS 1.3**.

2. Для серверного взаимодействия между `smart-home-service` и внешней платформой партнёра должна использоваться **взаимная аутентификация (mTLS)**
   либо эквивалентный механизм подтверждения доверенного сервиса.

3. Внешняя платформа не должна получать из систем PropDevelopment избыточные персональные данные. Необходимо применять принцип минимизации данных -
   передавать только данные, необходимые для выполнения функции управления домофоном и шлагбаумом.

4. Биометрические шаблоны лиц не должны храниться в контуре PropDevelopment без отдельного юридического и архитектурного обоснования. Распознавание
   лиц должно выполняться на стороне платформы партнёра. В системах PropDevelopment допускается хранение только:
    - Настроек доступа.
    - Идентификаторов пользователей.
    - Привязки пользователя к объекту недвижимости.
    - Списка разрешённых автомобилей.
    - Журнала событий доступа.

5. Номера автомобилей, разрешённых для проезда, должны храниться в БД `smart-home-db` в зашифрованном виде. Доступ к этим данным должен
   предоставляться только авторизованным сервисам.

6. Все команды управления устройствами и события от внешней платформы должны журналироваться. Журнал должен содержать:
    - Идентификатор пользователя.
    - Идентификатор объекта доступа.
    - Тип команды или события.
    - Дату и время.
    - Результат выполнения операции.

7. Должна быть реализована защита от повторной отправки запросов, подмены сообщений и несанкционированного вызова API с использованием:
    - Короткоживущих токенов доступа.
    - Проверки подписи сообщений или mTLS.
    - Ключ идемпотентности для критичных операций.

8. Внешняя платформа должна взаимодействовать только с выделенным сервисом `smart-home-service`. Прямой доступ к внутренним БД и сервисам
   PropDevelopment запрещён.

9. Необходимо обеспечить **tenant isolation** - пользователь, автомобиль или устройство не должны иметь доступа к данным другого жилого комплекса или
   управляющей компании.

10. Для защиты API должны использоваться механизмы:
    - Rate limiting.
    - Контроль аномальной активности.
    - Ограничение количества неуспешных запросов.
    - Уведомления о подозрительных действиях.

11. Все секреты интеграции (client secret, сертификаты, ключи подписи) должны храниться в защищённом хранилище секретов и не размещаться в коде или
    конфигурационных файлах приложений.

---

### 2. Протоколы аутентификации и авторизации

1. Для аутентификации пользователей мобильного приложения должен использоваться **OpenID Connect поверх OAuth 2.0**.

2. Для мобильного приложения рекомендуется использовать **Authorization Code Flow с PKCE**.

3. Для серверного взаимодействия между `smart-home-service` и внешней платформой должен использоваться:
    - **OAuth 2.0 Client Credentials Flow**.
    - Дополнительно **mTLS** для подтверждения доверенного канала связи.

4. Авторизация пользователя должна выполняться в два этапа:
    1. Проверка подлинности пользователя по токену доступа.
    2. Проверка прав пользователя на доступ к конкретному объекту (дом, подъезд, квартира, домофон, шлагбаум).

5. Токены доступа должны быть короткоживущими. Refresh token допускается только для мобильного приложения.

6. В токенах доступа должны передаваться только необходимые атрибуты:
    - Идентификатор пользователя.
    - Идентификатор роли.
    - Tenant (жилой комплекс / дом).
    - Разрешённые scopes.

7. Для API умного дома должны использоваться отдельные scopes, например:
    - `smart-home.read`
    - `smart-home.command`
    - `smart-home.events.read`
    - `smart-home.access.manage`

---

### 3. Организация взаимодействия между системами предприятия и внешней платформой

1. Пользователь проходит аутентификацию в мобильном приложении PropDevelopment через существующий контур аутентификации.

2. Мобильное приложение взаимодействует только с внутренним сервисом `smart-home-service`, не обращаясь напрямую к внешней платформе партнёра.

3. Сервис `smart-home-service` выполняет:
    - Проверку токена пользователя.
    - Проверку прав доступа к объекту недвижимости.
    - Проверку разрешённых сценариев доступа.
    - Проверку разрешённых автомобилей.

4. После успешной проверки `smart-home-service` обращается к API внешней платформы умного дома от имени доверенного сервиса PropDevelopment.

5. Внешняя платформа возвращает результат выполнения команды:
    - Доступ разрешён.
    - Доступ запрещён.
    - Устройство недоступно.
    - Ошибка обработки запроса.

6. Помимо команд управления должна поддерживаться передача событий от внешней платформы в `smart-home-service`, например:
    - Распознавание лица.
    - Распознавание номера автомобиля.
    - Успешное открытие двери.
    - Успешное открытие шлагбаума.
    - Отказ в доступе.
    - Состояние устройства.

7. Передача событий должна выполняться через защищённый webhook или API, защищённый mTLS и сервисной авторизацией.

8. `smart-home-service` сохраняет журнал событий и при необходимости передаёт агрегированные данные в DWH для аналитики.

9. В случае недоступности внешней платформы пользователь должен получать контролируемый ответ без раскрытия внутренней архитектуры системы.

10. Взаимодействие должно быть организовано по модели **internal facade**:
    - Мобильное приложение работает только с API PropDevelopment.
    - Вызовы к платформе партнёра выполняются через `smart-home-service`.
    - Внутренняя модель данных PropDevelopment не раскрывается внешней системе.
